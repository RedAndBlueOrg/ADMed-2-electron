# 핸드오프 페이로드 표준

> 이 문서는 메인 ↔ 서브에이전트 / 에이전트 ↔ 에이전트 핸드오프 시 본다. 무엇을 어떻게 전달하는지의 표준. 핸드오프 부실 = 결정 누락 = 재작업.

## 왜 표준이 필요한가

각 에이전트는 자기 영역 docs/ 만 read 한다 (격리). 그래서 **상위 에이전트가 결정한 사항이 명시적으로 페이로드에 담기지 않으면 다음 에이전트는 모른다**. 메인이 어설프게 요약하면 핵심 결정이 새고, 결과적으로 잘못된 구현 → 재작업.

## 표준 보고 페이로드 (모든 에이전트 공통)

각 에이전트의 보고 끝에 **반드시** 다음 두 섹션을 포함한다:

```
[핸드오프]
- 다음 단계: <에이전트 이름> 또는 "메인 결정 필요"
- 다음 에이전트가 알아야 할 결정사항 (≤5줄):
  - <결정 항목 1>
  - <결정 항목 2>
- 미결 / 사용자 확인 필요: <있으면 한 줄, 없으면 생략>

[갱신된 영속 자산]
- docs/<file>:<섹션> — <갱신 요약>
- impact-map 후보: <한 줄, 없으면 생략>
- incident-log 후보: <한 줄, 없으면 생략>
- progress.md 추가 후보: <한 줄, 없으면 생략>
```

메인은 다음 에이전트 호출 시 위 페이로드를 prompt 본문에 **그대로 인용**한다 (메인이 다시 요약 X).

## 페이로드 작성 원칙

1. **결정사항 ≠ 작업 결과** — "`playlist:prepare` 핸들러 추가했다"는 결과, "다운로드 진행률은 별도 IPC 이벤트(`download:progress`)로 push, invoke 응답엔 안 넣음 — 백그라운드 다운로드라서"는 결정. 다음 에이전트가 코드만 보면 모르는 것만 적는다.
2. **5줄 한도** — 길어지면 docs/ leaf 로 분리하고 페이로드엔 링크.
3. **미결은 명시적으로** — 추측해서 결정한 건 "추측: ~~ (사용자 확인 필요)" 라고 적어 다음 에이전트가 가정 맹신 회피.
4. **자산 갱신은 페이로드에 적힌 것만 신뢰** — 에이전트가 갱신했다고 적은 docs / impact-map / progress 만 메인이 신뢰 (실제 read 해서 검증 가능).

## 메인의 핸드오프 책임

- 페이로드를 prompt 에 그대로 인용 (요약 금지)
- "결정사항" 항목이 비어있으면 상위 에이전트에 다시 요청 (생략 금지)
- "미결 / 사용자 확인 필요" 항목이 있으면 다음 에이전트 호출 전에 사용자에게 먼저 확인

## 안티 패턴

- ❌ 메인이 에이전트 보고를 자기 말로 다시 요약해서 전달 (decision drift)
- ❌ "다음 단계: 알아서 진행" — 모호하면 메인이 명시적으로 정한다
- ❌ 결정사항 5줄 초과 — leaf 로 빼고 링크
- ❌ 자산 갱신을 페이로드에 안 적음 — 다음 에이전트가 동일 파일 또 갱신 → 충돌

## 핸드오프 검증 (메인 책임)

다음 에이전트 호출 직전 메인이 자체 점검:
- [ ] 페이로드의 "다음 단계" 가 명시됐는가
- [ ] "결정사항" 항목이 비어있지 않은가
- [ ] "미결" 항목이 있다면 사용자 확인을 거쳤는가
- [ ] 호출 prompt 에 페이로드를 그대로 인용했는가

이 4개가 통과돼야 다음 에이전트 호출.
