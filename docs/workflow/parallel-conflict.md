# 병렬 호출 vs 순차 — 충돌 처리

> 이 문서는 메인이 두 에이전트를 동시에 부를지·순차로 부를지 결정 시 본다. 충돌 발생 시 복구 절차 포함. 큰 정책은 [orchestration.md](orchestration.md).

## 왜 중요한가

병렬 호출은 효율적이지만 **같은 파일을 동시 수정하면 마지막 write 가 이긴다** (Edit 충돌, lock 없음).

## 병렬 OK 조건

- 두 에이전트가 **서로 다른 파일만 만짐**이 명백 (예: backend = `src/main/<모듈>.js`, frontend = `src/renderer/<모듈>.js` — 단 `preload.js` 와 `index.html` 은 양쪽이 만질 수 있으니 주의)
- 공유 표면(IPC 채널명, `preload.js` 브리지, `src/main/state.js` / `src/renderer/state.js` 공유 객체)을 **둘 다 읽기만** 함

## 순차로 강제

- 둘 다 같은 파일 / 같은 모듈 수정 가능성 (특히 `preload.js`)
- IPC 채널 신규·시그니처 변경 — backend(main) 가 채널·`ipcMain.handle`·`preload.js` 브리지 먼저 확정 → 그다음 frontend(renderer) 가 호출처·리스너 맞춤
- 한쪽 결정이 다른 쪽 prompt 에 페이로드로 들어가야 하는 경우
- 의심스러우면 **순차로**. 병렬은 충돌 없을 때만 가치.

## 결정 트리

```
backend(main) + frontend(renderer) 동시 필요?
  ├─ 아니오 → 단일 호출
  └─ 예 → IPC 채널 / preload.js / 공유 state 동시 수정 가능?
        ├─ 예 → 순차 (backend 가 채널·브리지 확정 → frontend 호출처)
        └─ 아니오 → 병렬 (한 turn 에 두 Agent 호출)
```

## 충돌 발견 시 복구 절차

병렬 호출 결과 양쪽 페이로드가 같은 파일에 변경 보고했을 때:

1. **현재 git 상태 확인**
   ```bash
   git status
   git diff <충돌 파일>      # 어느 변경이 살아남았는지
   ```
   먼저 끝난 쪽 변경은 거의 확실히 사라졌음 (Edit 가 덮어쓰기).

2. **사라진 변경 복구**
   - 해당 에이전트의 페이로드 `[갱신된 영속 자산]` + 보고 본문 read
   - 변경 의도가 작으면 (한두 줄): 메인이 직접 다시 적용
   - 크면: 그 에이전트를 다시 호출하며 prompt 에 "이전 변경이 충돌로 사라졌으니 다시 적용. 단, 다음 부분은 이미 다른 에이전트가 처리: <범위>" 명시

3. **두 변경이 의미상 호환 안 되면 (계약 충돌)**
   - 두 페이로드의 결정사항을 사용자에게 요약 제시 → 사용자가 우선 결정
   - 결정 후 한쪽 에이전트 재호출, prompt 에 사용자 결정 명시

4. **합쳐진 결과를 tester 로 검증** (충돌 복구 단계 자체가 새 변경 — 특히 IPC 3곳 동기화 재확인)

## 안티 패턴

- ❌ "둘 다 다른 파일 같으니 병렬" — 실제론 `preload.js` 나 공통 state 를 둘 다 수정한 경우 자주 발생
- ❌ "병렬이면 빠르니까 일단 병렬" — 충돌 복구 비용이 순차보다 큼
- ❌ 한쪽 페이로드 결정사항을 다른 쪽 호출 시 prompt 에 안 인용 (decision drift)
- ❌ 충돌 발견 후 사라진 변경을 메인이 짐작으로 재작성 (페이로드 read 필수)
